> **TL;DR：** 靠輪詢 /proc 收 metrics 的 agent，會持續不斷地吃 CPU。在同一台機器、收集同一組指標的前提下，改用 **eBPF** 把平均 CPU 開銷壓低了 **93.6%**、延遲也降了 **15.4%**。下面是我怎麼量的、結果長怎樣，以及什麼情況下你才真的需要在意這件事。

## 問題：你的監控正在默默燒 CPU

幾乎每一套線上系統都裝了監控 agent——`node-exporter`、cAdvisor、Datadog agent，或是某個前人留下、沒人敢動的自寫腳本。它們的共通點是：每隔幾秒就醒過來，讀一輪 `/proc`，把 CPU、記憶體、網路、磁碟的數字解析出來，再吐給後端。

這件事感覺「免費」，因為單看一次就只是讀幾個檔案。但監控的本質是**永不停歇**——它一天要跑上萬次，每台節點都跑，而且你跑的服務愈多、要收的 metrics 愈多，它就讀得愈勤。問題不是某一次讀檔很貴，而是這個成本是**長期、持續、隨規模放大**的。

弔詭的是，這幾乎沒人量過。大家會仔細優化自己的應用程式，卻把「觀測這件事本身的成本」當成理所當然的背景雜訊。所以我決定把它量出來——先量 `/proc` 輪詢到底吃掉多少，再看 eBPF 能不能做得更省。

## 你會學到

- `/proc` 輪詢式監控的開銷到底從哪裡來
- 怎麼「公平地」量這個開銷——一套你能自己重現的方法論
- eBPF 怎麼把它降下來、以及在什麼情境下差最多（什麼情境下其實沒差）

## 一般的做法：輪詢 /proc

`/proc` 是 Linux kernel 動態生成的虛擬檔案系統。它不是真的躺在磁碟上的檔案，而是你每次去讀 `/proc/stat`、`/proc/meminfo`、`/proc/<pid>/stat` 的當下，kernel 才即時把對應的核心資料結構序列化成文字給你。

監控 agent 的標準做法就是**定時輪詢（polling）**：起一個迴圈，每 1 秒（或 5 秒、15 秒）就把這一票 `/proc` 檔案全讀一遍、用字串解析撈出數字、算差值、送出去。看起來很單純，但成本藏在細節裡：

- **每次讀取都是一趟 syscall 來回**：`open` / `read` / `close`，每一趟都要從 user space 切進 kernel 再切回來，產生 context switch 與快取污染。
- **kernel 端要即時「生成」內容**：你讀的那一刻，kernel 才去走遍 task 結構、加總統計，把它格式化成文字。讀的檔案愈多、程序愈多，這個生成成本愈高。
- **user space 端要解析文字**：拿到的是一坨字串，agent 還得 parse、轉型、比對前一次的值。CPU 就這樣一點一點被吃掉。
- **它跟「有沒有事情發生」無關**：就算系統閒到發慌，輪詢還是照樣每秒醒來做一次完整的工。你付的是「持續輪詢」的固定成本，而不是「實際發生了多少事件」的成本。

在單機、低密度的環境，這些開銷小到可以忽略。但一旦進到高密度、多 pod、上百顆節點的叢集，每台都在每秒輪詢、每台都在解析文字，加總起來就不是雜訊了。

## eBPF 的做法：在 kernel 裡事件驅動

eBPF 把問題反過來想。與其「定時去問 kernel 現在怎麼樣」，不如「直接掛在 kernel 的事件上，有事才回報」。

你把一支經過 verifier 驗證、保證安全的小程式，掛到 kernel 裡的 tracepoint、kprobe 或 syscall 進入點上。當對應的事件真的發生時——一個程序被排程、一個封包進來、一次磁碟 I/O——你的 eBPF 程式就在 kernel 裡**就地**被觸發，直接讀到第一手的核心資料結構，不需要序列化成文字、也不需要 user space 一次次去 poll。

收集到的資料透過 **ring buffer** 高效地批次送到 user space。這跟輪詢有幾個本質差異：

- **沒有輪詢的固定開銷**：系統閒著時，幾乎沒有事件、也就幾乎沒有成本。成本跟著「實際事件量」走，而不是跟著「時鐘」走。
- **不用切上切下、不用解析文字**：資料在 kernel 裡就直接拿到結構化的數值，省掉了一輪輪的 syscall 來回與字串解析。
- **ring buffer 攤平了傳輸成本**：事件在 kernel 端緩衝、批次交給 user space，避免每個事件都各自付一次跨界成本。

## 結果

下面是同一台機器、同一組指標下，`/proc` 輪詢與 eBPF 兩種做法的對比：

| 指標 | /proc 輪詢 | eBPF | 改善幅度 |
|---|---|---|---|
| 平均 CPU 開銷 | 0.1323% | 0.0084% | **93.6%** |
| 最大 CPU 開銷 | 0.1781% | 0.0563% | **68.4%** |
| 平均延遲 | 3.9074 ms | 3.3053 ms | **15.4%** |

這張表的重點是：**最大的差距出現在 CPU 開銷上**。eBPF 把平均 CPU 開銷砍掉超過九成——因為它根本不做「持續輪詢」這件事，閒著時就幾乎不花成本。延遲的改善幅度（15.4%）相對溫和，這也合理：延遲本來就受系統其他因素影響較大，而省掉文字解析與多趟 syscall 帶來的，主要是更穩、更省的開銷曲線，而不是延遲的數量級差異。

✏️ **【需你補：數據解讀補充】** 如果你對這些數字有更具體的觀察（例如高負載時兩者差距是否拉得更開、CPU 開銷的變異度差異、是否有令你意外的點），建議補一兩句你自己的解讀——這是只有你跑過實測才知道的東西。

## 什麼時候該在意（什麼時候不必）

老實說，eBPF 不是萬靈丹，這個 93.6% 也不是「換了就一定爽」的保證。它真正發光的場景很明確：

- **高密度、多 pod 的節點**：一台機器上塞了幾十個容器、每個都要收 metrics 時，輪詢的成本是疊加的，eBPF 的事件驅動優勢會被放大。
- **大規模叢集**：上百、上千顆節點，每台省下來的那一點點 CPU 乘上節點數，就是實打實的成本與容量。
- **本來監控就吃很重的環境**：採樣頻率高、收集的 metrics 種類多時，輪詢的固定成本特別痛。

反過來說，這些情況下其實沒必要急著換：

- **小規模、單機或低密度**：絕對數字本來就很小（你看表上是 0.13% vs 0.008%），省下來的量在實務上感受不到，未必值得為它引入 eBPF 的複雜度與 kernel 版本相依。
- **團隊還沒準備好維運 eBPF**：CO-RE、BTF、kernel 相容性這些都是新的維運面，如果現有的輪詢方案夠用，貿然導入反而是負債。

一句話：**先量你自己的開銷，再決定要不要換**——而不是因為 eBPF 很潮就換。

## 重點整理

- **監控本身有成本，先量再說。** 大家都在優化應用，卻很少有人量過「觀測這件事」花了多少 CPU。
- **eBPF 是事件驅動，高密度環境省最多。** 它的省，來自「不輪詢」——閒著時幾乎不花錢，成本跟著事件量走而不是跟著時鐘走。
- **但要看規模，別為了用而用。** 小機器上的差異微乎其微；eBPF 的價值在規模放大後才顯現。
